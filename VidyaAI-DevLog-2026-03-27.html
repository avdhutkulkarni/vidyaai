1 · New API Route — /api/usage

What was built

New server-side endpoint app/api/usage/route.ts to fetch a student's doubt usage for today.

Logic

1
Verify Firebase Bearer token from Authorization header
2
Extract uid from decoded token
3
Read studentClass from query param (default: 10)
4
Call getUsageToday(uid, studentClass) from lib/doubtLimit.ts
5
Return: { used, limit, isPaid, boostDoubtsUsed, boostDoubtsRemaining }
Why: The doubt counter was initialised to 0 on page load and only updated after each question. Students would see wrong count until they asked their first doubt. Now the real count is loaded from server on app start.

Key detail

export const dynamic = 'force-dynamic' — required at top of every API route that reads live data. Prevents Next.js from statically caching the response.

3 · AI Personality — Simple English, Indian Examples

File: app/api/ask/route.ts — buildSystemPrompt()

What changed

Added a PERSONALITY block to the system prompt that runs for every conversation:

PERSONALITY RULES (added to system prompt)

  • Talk like a friendly elder brother explaining to a 10-year-old Indian child
  • ONLY English — no Hindi words, no Marathi words
  • Use real Indian examples: cricket, chai, mobile charging, auto rickshaw, mother cooking, mango season, railway station
  • Very simple words. Short sentences. Never sound like a textbook
  • Never say "therefore", "hence", "thus", "henceforth"
  • Say things like: "Think about it", "Here is the cool part", "Notice this", "Simple right?", "Now watch what happens"
Why: Previous version had Hindi casual words ("Arey", "dekh", "simple hai") which students from Maharashtra found confusing. The app is for Maharashtra HSC students — simple English is what they understand best in a study context.

Reply length updated

Changed from "max 4-5 lines" to "5-7 lines maximum" — slightly more explanation per reply while keeping it concise.

5 · Photo Upload Fix (Root Cause)

Bug

Photo upload appeared to work (file picker opened) but photo was never sent to AI.

Root cause

Inside handlePhoto(input), the first line inside reader.onload was:

document.getElementById('ch-hdr').classList.add('hidden')

The chat header element had class="ch-hdr" but no id attribute. So getElementById('ch-hdr') returned null. Calling .classList on null throws a TypeError, which silently killed the entire callback — the fetch('/api/ask') call was never reached.

Fix 1: Added id="ch-hdr" to the div in app.html: <div class="ch-hdr" id="ch-hdr">
Fix 2: Added null guard: const hdr=document.getElementById('ch-hdr'); if(hdr)hdr.classList.add('hidden')

Photo menu — mobile compatibility fix

The menu items used <div onclick="document.getElementById('photo-input').click()">. Programmatic .click() on file inputs is unreliable on mobile browsers.

Fix: Replaced <div onclick> with <label for="photo-input"> and <label for="camera-input">. A <label> directly activates its linked file input — the correct semantic approach, works on all browsers.

Added error handling

  • reader.onerror handler — shows toast if file read fails
  • try/catch inside reader.onload — catches any unexpected errors
  • 401 auth error handler in the photo fetch chain
  • Null check for document.getElementById('msgs')

7 · New Numerical Conversation Flow

File: app/api/ask/route.ts

Applies to: numerical questions only

Old flow (3 steps)

Hint 1 (concept + formula) → Hint 2 (first step) → Full Solution

New flow (5 steps)

1
Hook question — real-life Indian example, 1-2 lines only
2
Hint 1 — concept name + 2-3 line simple explanation with Indian example
3
Hint 2 — formula + what each variable means (1-2 lines). Then asks: "Want complete solution? Yes / No"
4
Full point-wise solution — Step 1, Step 2, Step 3, ∴ Answer [units]. Then shows two buttons: "Want to go deeper?" and "Want similar questions?"
5a
Go deeper — detailed concept explanation, still simple, Indian examples, max 8 lines
5b
Similar questions — AI outputs 2 questions in exact format (parsed by frontend and saved to Firebase)

Practice questions format (AI must use exactly)

📝 Practice Q1 (Similar): [full question with all values and units]

📝 Practice Q2 (Tricky): [harder variation of same concept]

[PMETA]concept:[concept name]|subject:[subject name][/PMETA]
Why PMETA tag: The frontend needs concept and subject to save questions to Firebase. The AI tags them directly in the response so the app doesn't have to guess from context. The [PMETA] block is stripped before displaying to the student.

8 · Tappable Action Buttons

File: public/app.html — formatReply() function

Problem

The end-of-reply line ✅ Samjhla! | 🔄 Nahi samjhla | 💡 Exam tip chahiye? was rendered as plain text. Students couldn't tap it.

Solution

Added regex patterns in formatReply() to convert AI output to interactive buttons:

AI outputsRenders asFunction called
✅ Got it! | 🔄 Still confused | 💡 Exam tip?3 tap buttons (row)doSamjhla / doNahiSamjhla / doExamTip
Want complete solution? Yes / NoYes/No buttonswantCompleteSolution / wantNo
Want to go deeper? Yes / NoYes/No buttonswantDeeper / wantNo
Want similar questions? Yes / NoYes/No buttonswantSimilar / wantNo

📝 Practice Q2 (Tricky): …Tricky card (red border) + auto-savesavePracticeQuestion()

Button functions logic

  • doSamjhla(btn) — disables row, resets thumbsDownCount=0, shows "✓ Got it!" XP toast
  • doNahiSamjhla(btn) — increments thumbsDownCount, sets message, calls send()
  • doExamTip(btn) — sets message "Give me an exam tip for this concept", calls send()
  • wantCompleteSolution / wantDeeper / wantSimilar — disable row, set appropriate message, call send()

Practice questions → Firebase

savePracticeQuestion(question, type, concept, subject) — POST to /api/practice. Saved to practiceQuestions/{uid}/questions/{auto-id} with fields:

  • question, type (similar/tricky), concept, subject, studentClass
  • solved: false, uid, createdAt
Future use: These questions will power the Jigyasa/Recharge Mode where students can practice saved questions.

9 · fbBad "Still Confused" Fix

File: public/app.html

Bug

fbBad(btn) was called when student tapped "👎 Still confused". It changed the button text to "Trying a different approach…" but never sent anything to the AI. No new explanation appeared.

Root cause

Previous fix used setTimeout(()=>send(), 300) which could fail silently if any error occurred in send() before the fetch (null checks, element not found, etc.).

Fix

Rewrote fbBad() as a self-contained direct API caller — same pattern as the working photo upload. No dependency on send(), no textarea, no setTimeout race.

New logic:
1. Increment thumbsDownCount
2. Change button text
3. Get box = document.getElementById('msgs') directly
4. Create aiDiv with "Let me try a different way… 🤔"
5. Push message to conversationHistory
6. Call fetch('/api/ask') with thumbsDownCount already incremented
7. Handle 429 / 401 / network errors inline
Key insight: The AI system prompt checks thumbsDownCount. At count=1 it uses a "completely different real-life analogy". At count≥2 it uses "the simplest possible everyday Indian example". So incrementing before the API call is critical.

11 · Admin Panel + User Approval System

Problem

Any Google account in the world could log in and use VidyaAI. No access control existed at the user level.

Architecture

New API Routes

  • app/api/auth/check/route.ts — POST: verify token, create/check user record
  • app/api/admin/users/route.ts — GET: list users, POST: approve/reject/revoke

New Pages

  • app/admin/page.tsx — Full admin panel UI
  • Updated app/vidyaai-login/page.tsx
  • Updated app/page.tsx
  • Updated middleware.ts

Student login flow (new)

1
Student taps "Sign in with Google" → Google OAuth popup
2
App calls POST /api/auth/check with Firebase ID token
3
New student: Creates users/{uid} with approved: false, status: 'pending'
4a
Approved: Sets cookie, redirects to /vidyaai
4b
Not approved: Signs out, redirects to /?reason=pending → shows "Waiting for Approval" screen

Admin panel at /admin

Protected by middleware (requires auth cookie) + ADMIN_UIDS check on every API call.

TabShowsActions
PendingNew students awaiting approval (count badge in red)✓ Approve, ✗ Reject
ApprovedActive studentsRevoke (moves back to pending)
RejectedRejected studentsRe-approve
All UsersEveryoneAll actions

Info shown per student

Name, Email, Class, Plan (Free / ⚡ Boost), Total doubts asked, Joined date, Last active (relative time)

Paid users — auto-approval

When a student pays and plan: 'boost' is set in Firestore, /api/auth/check automatically sets approved: true on their next login. Admin can still manually revoke if needed.

Server-side enforcement

All 4 main API routes now check approved: true in Firestore after token verification:

  • app/api/ask/route.ts
  • app/api/usage/route.ts
  • app/api/practice/route.ts
  • app/api/doubt-history/route.ts
Even with a valid Firebase token, an unapproved user gets 403 NOT_APPROVED. UI bypass is impossible.

One student = one Google account

Firebase Auth ties each login to a Google account's unique UID. The approval is stored per UID. Students cannot share accounts (they'd need to share the Google password). If a student logs in with a different Google account, it creates a new pending record — admin only approves the correct one.

Setup required

Add your Firebase UID to Vercel environment variables:
ADMIN_UIDS = your-firebase-uid-here
Get your UID from: Firebase Console → Authentication → Users → click your account.
